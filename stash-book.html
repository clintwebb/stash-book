Introduction

Stash was specifically designed to provide a data storage system suitable for modern websites that need to quickly provide a large amount of small chunks of data.

Websites are normally developed very fluidly. The data storage system needs to be flexible, and easy to use.

Goals of the Stash Project

When designing and developing Stash, there were some specific goals in mind:

  • 100% uptime - Aside from undetected bugs, the database needs to be up all the time, even when changes need to be made to the database software and libraries themselves.
  • Automatic Replication - Changes made on the master automatically get replicated to slave servers.
  • Master and slave ignorant - The client connects to a server, but doesn't know or care if it is a master or a slave. If it makes any requests that only a master can do, the slave will send the request on.
  • Type Agnostic - Didn't want to be tied to a particular data-type for a column, if we decide that we need to store a string instead of an integer, we can do so.

Different to SQL

Stash is a lot different to MySQL, Oracle and other SQL-like databases.

Different to other Non-SQL systems.

Features

The AUTO value. In SQL it is common to allow key columns to auto-increment the value. This is a useful unique ID generator. With Stash, you can indicate that an automatic number be generated. THe advantage is that when using a single table with content-named rows, you can generate a unique id on the key, even though different types of data is being stored. This allows a certain flexibility that was not possible before.
